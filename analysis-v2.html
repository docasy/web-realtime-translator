2. 流畅度瓶颈：为什么感觉"卡"

2.1 VAD 问题 —— 固定阈值 + 无自适应

🔴 P0：固定 VAD 阈值 (0.5)

当前 silero-vad 使用硬编码的 0.5 阈值。在安静环境中偏高（漏掉轻声说话），在嘈杂环境中偏低（误触发）。
没有噪声底噪追踪，没有自适应调节。
后果：安静时用户说完整句话才开始识别（感觉"没反应"）；嘈杂时疯狂误触发（CPU 被无效 ASR 占满）。

🟡 P1：ScriptProcessorNode 主线程阻塞

ScriptProcessorNode 运行在浏览器主线程。任何重计算（大数据量重采样、大 JSON 序列化）会阻塞 UI 渲染，导致录音按钮卡顿、气泡动画掉帧。
应迁移到 AudioWorklet（专用音频线程），但需要独立 JS 文件。

2.2 ASR 问题 —— 模型 + 无预处理

🔴 P0：无音频预处理

原始麦克风音频直接喂给 Whisper，没有噪声门控（noise gate）、没有响度归一化（loudness normalization）。
背景噪音被当作语音转写，产生大量乱码文本，浪费翻译 API 调用。
后果：误识别率高，翻译出无意义内容。

🟡 P1：whisper base 在 CPU 上的硬限制

base 模型在 CPU 上处理 5 秒音频约需 1-3 秒。换成 small 更准但更慢，tiny 更快但不准。
在不换硬件的前提下，无法根本性突破这个限制。预处理能减少无效 ASR 调用，但单次调用延迟不变。

2.3 翻译问题 —— 无上下文 + 无缓存

🔴 P0：翻译无对话上下文

当前每次翻译请求是独立的——LLM 不知道上一句说了什么。对话翻译的连贯性依赖上下文：
"他来了" → "He's here"，但下一句"让他进来" → "Let him in" 需要有"他"的指代认知。
没有上下文时，翻译容易出现指代断裂、语气不一致。

🔴 P0：无翻译缓存

口语对话中高频短语反复出现（"好的"、"等一下"、"你说什么"）。每次重新调用 API，浪费时间和额度。
LRU 缓存可以消除 15-25% 的重复翻译请求。

4. 改进方案：按优先级排序

🔴 P0-1：自适应 VAD 阈值 + 噪声底噪追踪

实现：在 VAD 循环中维护一个指数移动平均的噪声底噪估计（noise floor）。当检测到长时间静音时更新底噪。实际阈值 = max(0.3, min(0.7, noise_floor + 0.15))。
效果：安静环境更灵敏（不错过轻声），嘈杂环境更稳定（不误触发）。

🔴 P0-2：音频噪声门控预处理

实现：在 WAV 编码前，对 speech_buffer 做简单的 RMS 能量检测。低于 -40dB 的帧视为静音并衰减 6dB。
效果：减少静音帧进入 Whisper，提升识别准确率，减少乱码输出。

🔴 P0-3：翻译对话上下文 + LRU 缓存

实现：维护最近 5 句话的"原文+译文"对列表，注入翻译 system prompt 作为上下文。同时用 dict 做简单 LRU 缓存（最大 200 条），命中时直接返回。
效果：指代连贯性显著提升；重复短语 0 延迟返回。

🟡 P1-1：VAD 段首回溯

当前 VAD 检测到语音开始时，前面的帧已经被丢弃了。增加一个 5 帧的预滚动缓冲区（pre-roll buffer），确保语音开头不被截断。

🟡 P1-2：翻译 prompt 加入语言特定规则

当前 prompt 对所有语言对通用。针对具体语言对（如 fr→zh）加入特定规则：法语否定式、代词前置、时态对应等。

🔵 P2-1：AudioWorklet 替代 ScriptProcessorNode

ScriptProcessorNode 已废弃，运行在主线程。AudioWorklet 在专用音频线程运行，不会阻塞 UI，支持更大的 buffer 和更稳定的时序。

🔵 P2-2：whisper base → small 或 large-v3

base 是速度和准确率的折中。small (500MB) 准确率高 15-20%，CPU 上延迟多 1-2 秒。large-v3 (3GB) 最准但需 GPU。

性价比排序：自适应 VAD > 噪声门控 > 翻译缓存 > 对话上下文 > VAD 预滚动 > AudioWorklet > 换 Whisper 模型。
前三项改动量约 50 行代码，对体感提升最明显。
